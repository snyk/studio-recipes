# Snyk Security Hooks

A pre-commit hook system that prevents developers from committing code that introduces new security vulnerabilities.

## Features

- **SAST Protection**: Blocks commits with new code vulnerabilities (XSS, SQL injection, path traversal, etc.)
- **SCA Protection**: Blocks dependency changes that increase security risk
- **Smart Filtering**: Only blocks on NEW vulnerabilities, not pre-existing issues
- **Background Pre-Scanning**: Scans files on save for instant commit checks
- **Intelligent Caching**: Avoids re-scanning unchanged code
- **Copy-Paste Fix Commands**: Outputs ready-to-use commands for your AI assistant
- **Works with AI Code Generation**: Catches vulnerabilities whether coded manually or generated by AI

## Quick Start

```bash
# 1. Clone the studio-recipes repository (one time)
git clone https://github.com/snyk/studio-recipes.git

# 2. Run the installer from your project
cd your-project
bash ../studio-recipes/guardrail_directives/secure_at_inception/kiro_hooks/install.sh

# 3. Code as normal - hooks run automatically
# - Kiro: Scans on file save (background)
# - Git: Scans on commit (uses cached results)

git add .
git commit -m "my changes"

# 4. If blocked, copy the fix command into your AI assistant
/snyk-fix-batch javascript/PT@server.ts:45, sca:lodash
```

## Installation

### Step 1: Clone studio-recipes (one time)

```bash
# Clone to a location accessible to all your projects
cd ~/repos  # or wherever you keep repos
git clone https://github.com/snyk/studio-recipes.git
```

### Step 2: Run the installer in your project

The installer sets up both git pre-commit hooks and Kiro background scanning:

```bash
# From your project directory
cd my-project

# Run the installer (adjust path to where you cloned studio-recipes)
bash ../studio-recipes/guardrail_directives/secure_at_inception/kiro_hooks/install.sh

# Or with full path
bash ~/repos/studio-recipes/guardrail_directives/secure_at_inception/kiro_hooks/install.sh
```

**What gets installed:**
- Git pre-commit hook → `.git/hooks/pre-commit`
- Hook libraries → `.git/hooks/lib/*.py`
- Kiro background scanner → `.kiro/hooks/kiro_background_scanner.py`
- Kiro hook configuration → `.kiro/hooks/background-security-scan.kiro.hook`

**Chaining with existing hooks:**
If you already have a pre-commit hook, the installer will ask if you want to:
1. Replace it (backs up to `pre-commit.backup`)
2. Chain both hooks (runs both)
3. Cancel installation

## How It Works

### Architecture Overview

```
┌─────────────────────────────────────────────────────────────────────────┐
│                  BACKGROUND SCANNING (Kiro Hook)                        │
│                  Runs on every file save                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   File Saved ──▶ Debounce (2s) ──▶ Background Scan ──▶ Cache Result     │
│                                                                         │
│   • Non-blocking - you keep coding                                      │
│   • Results stored in /tmp/snyk-cache-{hash}/                           │
│   • Notification if issues found                                        │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                  GIT PRE-COMMIT HOOK                                    │
│                  Runs on every commit                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   1. Analyze staged changes (files, packages)                           │
│                                                                         │
│   2. Check cache for each file/package                                  │
│      ├── Cache HIT  → Use cached results (~0s)                          │
│      └── Cache MISS → Run fresh scan (~30-60s)                          │
│                                                                         │
│   3. Filter to NEW vulnerabilities only                                 │
│      • Code: Is vuln line in modified range?                            │
│      • SCA: Does new version have more critical/high?                   │
│                                                                         │
│   4. Decision                                                           │
│      ├── NEW issues found → BLOCK commit                                │
│      └── No new issues → ALLOW commit                                   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### Performance Comparison

| Scenario | Without Cache | With Cache |
|----------|--------------|------------|
| 5 files, all cached | 45-90s | **<1s** |
| 5 files, 2 uncached | 45-90s | **~15s** |
| First commit (cold) | 45-90s | 45-90s |

## Blocking Logic

### Code Vulnerabilities (SAST)
**Block if**: ANY new vulnerability is introduced in modified lines

A vulnerability is considered "new" if:
- The file was modified in this commit, AND
- The vulnerability's line number falls within a modified range

### Dependency Vulnerabilities (SCA)
**Block if**: Package change results in MORE critical or high severity vulnerabilities

| Scenario | Result |
|----------|--------|
| Old: 0 crit, 0 high → New: 0 crit, 1 high | ❌ BLOCK |
| Old: 1 crit, 0 high → New: 0 crit, 2 high | ✅ ALLOW (net improvement) |
| Old: 1 crit, 1 high → New: 1 crit, 2 high | ❌ BLOCK |
| Old: 2 crit, 0 high → New: 1 crit, 0 high | ✅ ALLOW (improvement) |
| New package with 1 critical | ❌ BLOCK |
| New package with only medium/low | ✅ ALLOW |

**Note**: Medium and low severity issues are ignored for the blocking decision.

## When Blocked

When the hook blocks your commit, it displays:

```
╔══════════════════════════════════════════════════════════════╗
║           HOW TO FIX THESE ISSUES                            ║
╚══════════════════════════════════════════════════════════════╝

Copy and paste this command into your AI assistant:

┌────────────────────────────────────────────────────────────────┐
│  /snyk-fix-batch javascript/PT@server.ts:45, sca:lodash        │
└────────────────────────────────────────────────────────────────┘
```

Copy the `/snyk-fix-batch` command into Cursor and your AI assistant will:
1. Fix each vulnerability
2. Re-scan to verify fixes
3. Optionally create a PR

## Caching System

### Cache Location
Cache is stored in the **system temp directory** to avoid project pollution:
```
/tmp/snyk-cache-{workspace_hash}/
├── sast/
│   └── {filename}.{content_hash}.json
├── sca/
│   └── {package}@{version}.json
├── meta.json
└── notifications.json
```

The `{workspace_hash}` is derived from your project path, ensuring different projects have isolated caches.

**Benefits of temp directory:**
- ✅ No project directory pollution
- ✅ No `.gitignore` entry needed
- ✅ Automatic cleanup on system reboot
- ⚠️ Cache is cleared on reboot (cold start after restart)

### Cache Behavior
- **SAST**: Keyed by `file + content_hash` - invalidates when file changes
- **SCA**: Keyed by `package@version` - invalidates when version changes
- **TTL**: 12 hours (configurable)
- **Isolation**: Each project has its own cache via workspace hash

### Cache Management
```bash
# View cache statistics for current project
python3 kiro_hooks/git/lib/cache.py --stats

# View cache for a specific workspace
python3 kiro_hooks/git/lib/cache.py --stats --workspace /path/to/project

# Clear all cache entries
python3 kiro_hooks/git/lib/cache.py --clear

# Remove only expired entries
python3 kiro_hooks/git/lib/cache.py --cleanup

# Find your cache location
python3 -c "from kiro_hooks.git.lib.cache import get_cache_dir; print(get_cache_dir())"
```

## Configuration

### Environment Variables

| Variable | Default | Description |
|----------|---------|-------------|
| `SNYK_HOOK_DEBUG` | `0` | Set to `1` for verbose output |
| `SNYK_HOOK_QUICK` | `0` | Set to `1` to skip old version comparison (faster) |
| `SNYK_HOOK_NO_CACHE` | `0` | Set to `1` to disable cache (force fresh scans) |

### Examples
```bash
# Run with debug output
SNYK_HOOK_DEBUG=1 git commit -m "test"

# Disable cache for this commit
SNYK_HOOK_NO_CACHE=1 git commit -m "test"
```

## Kiro Background Scanner

The background scanner is **automatically installed** by the installer and configured to run on file saves.

**How it works:**
1. You save a file in Kiro
2. Scanner waits 2 seconds (debouncing)
3. Launches background scan via `scan_worker.py`
4. Results cached to `/tmp/snyk-cache-{hash}/`
5. Next commit uses cached results (instant!)

**File patterns monitored:**
- Code: `*.js`, `*.ts`, `*.py`, `*.java`, `*.go`, etc.
- Packages: `package.json`, `requirements.txt`, `pom.xml`, etc.

**Logs:**
- Scanner activity: Check Kiro's output panel
- Scan worker: `/tmp/snyk-scan-worker.log`

## Bypassing the Hook

In emergencies, you can bypass the hook:

```bash
git commit --no-verify -m "emergency fix"
```

**⚠️ Use sparingly**: This defeats the purpose of the security gate.

## Troubleshooting

### "Snyk CLI not found"
```bash
# Install Snyk CLI globally
npm install -g snyk
snyk auth
```

### "Snyk not authenticated"
```bash
snyk auth
```

### Slow commits (cache not working)
Check if cache is populated:
```bash
python3 .git/hooks/lib/cache.py --stats
```

If cache is empty, ensure:
1. Kiro background scanner is running (check `.kiro/hooks/`)
2. Files are being saved in Kiro (triggers background scans)
3. Wait 2+ seconds after save for debounce to trigger scan

### Force fresh scan (skip cache)
```bash
SNYK_HOOK_NO_CACHE=1 git commit -m "test"
```

### View background scan logs
```bash
cat /tmp/snyk-scan-worker.log
```

### Hook errors but doesn't block
The hook is designed to fail-open. Check the error message and:
1. Ensure Snyk CLI is working: `snyk code test .`
2. Check Python version: `python3 --version` (need 3.8+)
3. Enable debug mode: `SNYK_HOOK_DEBUG=1 git commit -m "test"`

### Uninstalling
```bash
# Remove the git hook
rm .git/hooks/pre-commit
rm -rf .git/hooks/lib

# Remove Kiro hooks
rm -rf .kiro/hooks/

# Clear cache (optional - it's in temp anyway)
python3 .git/hooks/lib/cache.py --clear
```
